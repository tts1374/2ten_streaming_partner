# AITuber配信パートナー Ph1 要求定義

## 1. 目的

Ph1の目的は、YouTube Liveを主対象にした「ローカル実演可能なAITuber共同MC」を作ること。

ライブチャット、音声入力、無入力時の話題生成をトリガーに、ローカルLLMでコメント選定と回答生成を行い、TTS音声とOBS字幕で配信に出力する。

## 2. 前提

- 対象マシン
  - OS: Windows
  - CPU: Intel Core i5-14400F
  - GPU: NVIDIA RTX 4060 Ti
  - Memory: 32GB
- LLM実行基盤
  - Ollamaを前提とする。
  - リアルタイム分類、安全判定、コメント抽出は`qwen3.5:4b`を既定候補とする。
  - 通常コメントに対するリアルタイム回答生成は`qwen3:8b`を既定候補とする。
  - OBSキャプチャ画像など画像付き入力がある場合のみ、vision対応の`qwen3.5:9b`へ切り替える。
  - `qwen3.5:9b`はVRAM 8GB環境ではCPU/GPU分割実行になりやすいため、常時リアルタイム経路には入れない。
  - ELYZA Llama3 8Bは、Ph1では日本語の自然さ、キャラクター口調、将来の長期記憶レビュー検証用の比較モデルとして扱う。
  - Qwen系モデルは配信用途では`think: false`を指定し、thinking出力を抑制する。
  - モデル切替時のロード待ちを避けるため、リアルタイム経路で使うモデルは`keep_alive`により一定時間常駐させる。
- STT
  - ローカルWhisper系を前提とする。
  - 実装候補はfaster-whisper系。
- TTS
  - Ph1の既定候補はAivisSpeech。
  - Qwen3-TTSはボイスクローンや表現力の将来検証候補とし、Ph1必須依存にはしない。
- 記憶・検索
  - 履歴DBに加えてベクトル検索まで扱う。
  - SQLiteとLanceDBの組み合わせを初期候補とする。

## 3. 機能要求

### 3.1 入力トリガー

- YouTube Live Chatを一定間隔で取得する。
- YouTube Live Chat取得では、`activeLiveChatId`の直接指定に加えて、配信の動画IDから起動時に`activeLiveChatId`を解決できる。
- YouTube Live Chat取得では、新規コメントの正規化、メッセージIDによる重複排除、APIが返す推奨ポーリング間隔の尊重を行う。
- 起動直後に返る過去コメントは既定で読み飛ばし、配信中の新規コメントから反応を開始できる。
- YouTube Live Chat APIの代表的な停止理由、期限切れpage token、終了済み配信、チャット無効、権限やquota問題は、運用者が次の対応を判断しやすいエラーとして表示する。
- 新規コメントがある場合、回答候補コメントを1件または少数抽出する。
- YouTube Chatが短時間に複数件返った場合は、すべてに反応せず、設定された上限件数まで軽量選定してから回答経路へ流す。
- YouTube Chat入力では、長すぎるコメント、記号比率が高いコメント、同一poll内および短時間内の同文重複、単一文字の反復が多いコメントを事前に除外できる。
- マイク音声を文字起こしし、音声入力イベントとして扱う。
- チャット・音声入力が一定時間ない場合、音楽ゲーム配信向けの自然な話題を生成する。
- idle topicは連発を避けるため、通常入力後に最初の話題を出すまでの待機時間と、idle topic後に次のidle topicを出すまでの待機時間を別々に設定できる。
- OBSキャプチャ画像添付はPh1では任意拡張扱いとし、将来追加可能なインターフェースだけ考慮する。

### 3.2 コメント選定

- コメント抽出はローカルLLMで実施する。
- 抽出目的は以下とする。
  - 問題のあるコメントを回避する。
  - 話題が広がりやすいコメントを選ぶ。
  - 配信の流れに合うコメントを優先する。
- コメント候補は、安全性、話題性、文脈適合度、返答しやすさで評価する。
- Ph1初期のYouTube Chat選定は、モデレーター、配信者、質問、音量・表示・遅延など運用確認系コメントを軽く優先する決定的スコアリングから始め、将来LLM選定へ拡張できるようにする。
- 荒らし対策は、YouTube Chat入力段の軽量フィルタ、入力安全判定、出力安全判定の多段構えとする。

### 3.3 安全設計

- 以下のコメントは回答対象から除外、または話題転換する。
  - 個人情報を含むもの。
  - 差別、性的、暴力、違法誘導を含むもの。
  - 荒らし、過度な煽り、炎上誘発を目的とするもの。
  - 配信者、視聴者、第三者を攻撃するもの。
- 回答生成前にも安全チェックを行う。
- 危険または扱いにくい入力は、以下のいずれかに分類する。
  - 拾わない。
  - 軽く流す。
  - 一般化して安全な話題へ転換する。

### 3.4 回答生成

- ローカルLLMで回答を生成する。
- 通常コメントへの回答生成は`qwen3:8b`を使用する。
- コメント選定、安全判定、入力分類は`qwen3.5:4b`を使用する。
- 画像付き入力がある場合のみ`qwen3.5:9b`で画像内容を解析し、解析結果を回答生成の補助文脈として扱う。
- 画像解析は配信の発話テンポを阻害しないよう、原則として非同期または低頻度で実行する。
- ELYZA Llama3 8Bはリアルタイム発話経路には常時入れず、人格・日本語品質レビュー、ログ比較、将来の長期記憶レビュー候補として扱う。
- 回答は共同MCとして短めに返す。
- 音楽ゲーム配信に合う、明るくフランクな口調にする。
- 堅苦しすぎる説明調は避ける。
- 人間配信者の主導権を奪わず、コメント拾い、話題展開、リアクション、場の整理を担う。

### 3.5 出力

- 生成回答をTTSで音声出力する。
- OBSにはブラウザソース用字幕オーバーレイを提供する。
- 字幕には現在発話中のテキストと簡易状態を表示する。
- Ph1初期の字幕オーバーレイは、最小HTMLとSSEで`OverlayState`を受け取り、字幕firstの薄いUIとして表示する。
- 字幕は発話終了後に短い余韻を置いて消し、初期値は2.5秒とする。
- Live2D、3Dモデル、口パク、表情制御はPh2以降の候補とする。

### 3.6 保存・記憶

- SQLiteに以下を保存する。
  - 入力イベント。
  - 採用コメント。
  - 安全判定。
  - 生成回答。
  - TTS結果。
  - 配信セッション。
- LanceDBに以下を保存する。
  - 過去会話の埋め込み。
  - 配信内文脈。
  - キャラ設定や運用メモの検索用データ。
- Ph1では、過去の関連発言を数件参照する程度に留める。
- 長期記憶の自動更新は暴走しやすいため、明示ルールを設ける。
- ローカル実機確認後に、SQLiteへ保存された入力イベント、生成回答、TTS結果、LLM呼び出しをCLIで確認できる。

## 4. キャラクター要求

- 初期キャラは「相棒・進行役」とする。
- 人間配信者との関係性は「共同MC」とする。
- 音楽ゲーム配信を主な活動領域とする。
- 口調は明るくフランクにする。
- ただし、過度な煽りや攻撃的なツッコミは避ける。
- 主な役割は以下とする。
  - コメントを拾う。
  - 話題を広げる。
  - 配信者のプレイや発言にリアクションする。
  - 場が止まったときに自然な話題を出す。
  - 荒れそうな話題を広げず、穏当に流す。

### 4.1 入力別の会話目的

- 音声入力は人間配信者の発言として扱い、人間配信者と直接会話する。
- YouTube Chat入力は視聴者コメントとして扱い、コメント内容を人間配信者に取り次いだうえで、話題が広がる場合は人間配信者に向けて短く広げる。
- idle topicは単独の独り言ではなく、直近のYouTube Chatまたは音声入力に関連した話題を人間配信者へ投げかける。
- 人間配信者の呼び名は設定で変更でき、既定値は「つてん」とする。

## 5. 初期インターフェース案

### 5.1 InputEvent

- `source`: `youtube_chat | voice | idle_topic`
- `text`
- `author`
- `timestamp`
- `metadata`
- `image_ref` optional

### 5.2 SafetyDecision

- `status`: `allow | ignore | deflect | block`
- `reasons`
- `safe_topic`
- `confidence`

### 5.3 SelectedPrompt

- `input_event_id`
- `selection_reason`
- `topic_potential`
- `safety_decision`

### 5.4 GeneratedReply

- `text`
- `persona_version`
- `memory_refs`
- `generation_model`
- `latency_ms`

### 5.5 SpeechJob

- `reply_id`
- `tts_engine`
- `voice_id`
- `audio_path`
- `status`

### 5.6 PersonaProfile

- 共同MC。
- 相棒・進行役。
- 音楽ゲーム配信向け。
- 明るくフランク。
- 配信者を立てる。
- 荒れそうな話題を広げない。

## 6. テスト観点

- YouTube Live Chatの新規取得、重複排除、最適コメント抽出が動くこと。
- 荒らし、個人情報、差別的コメントが回答対象から除外されること。
- マイク音声を文字起こしし、回答生成まで流れること。
- 無入力が続いた場合、音楽ゲーム配信向けの自然な話題が生成されること。
- 生成回答がAivisSpeechで音声化されること。
- OBS字幕に生成回答と同じテキストが表示されること。
- SQLiteとLanceDBに履歴が保存されること。
- 直近または関連する過去文脈を回答生成に参照できること。
- RTX 4060 Ti環境で、チャット入力から字幕・音声開始までの体感遅延が配信運用上許容できること。
- `qwen3.5:4b`、`qwen3:8b`、`qwen3.5:9b`、ELYZA Llama3 8Bの役割境界がログに保存され、どのモデルがどの判断を行ったか追跡できること。
- Qwen系モデルの呼び出しで`think: false`が指定され、thinking出力が字幕やTTSに流れないこと。
- 画像付き入力がない通常コメントでは`qwen3.5:9b`が呼ばれないこと。

## 7. Ph1の範囲外

- 実配信の完全自動運用。
- Live2Dまたは3Dモデル制御。
- 口パク、表情制御。
- 高度なOBSシーン制御。
- Qwen3-TTSの本格導入。
- 長期記憶の自動人格反映。
- `qwen3.5:9b`を通常コメント回答の常時リアルタイム経路に入れること。
